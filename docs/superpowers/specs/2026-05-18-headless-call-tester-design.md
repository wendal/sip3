# SIP3 Headless Call Tester Design

## Goal

Add a repository-owned headless SIP call tester that can reliably reproduce and catch TLS phone-to-phone regressions in SIP3. The first version must turn the current manual Linphone checks into repeatable automated scenarios for TLS registration, MESSAGE delivery, call setup, bidirectional RTP reachability, and clean call teardown.

## Confirmed decisions

- Shape: repository-owned Rust headless test client, not a GUI app.
- Primary scope: TLS phone-to-phone flows that model two Linphone-like SIP endpoints.
- Success criteria for calls: verify bidirectional RTP packet flow; do not validate decoded audio content in v1.
- Runtime targets: support both local/CI against Docker Compose and probe-style runs against a real SIP domain.
- First version remains focused on signaling and media-path correctness for plain SIP/TLS calls.
- Out of scope for v1: WebRTC, video, audio quality analysis, real audio decoding, and UI.

## Architecture

The tester is implemented as an internal Rust tool in the backend workspace so it can share the project toolchain and evolve alongside SIP3. It is not embedded into ordinary unit tests; instead, it is an explicit executable used by local developers, CI jobs, and optional production-adjacent probes.

The design has five focused components:

1. `SipEndpoint` — one headless endpoint with TLS connection management, REGISTER, MESSAGE, INVITE, ACK, BYE, CANCEL, and incoming SIP event handling.
2. `DialogController` — the per-call state machine that drives invite setup, answer handling, acknowledgment, teardown, and timeout decisions.
3. `RtpProbe` — a minimal RTP sender/receiver that binds UDP ports, emits fixed-interval RTP packets, and counts inbound packets from the negotiated peer.
4. `ScenarioRunner` — orchestration for end-to-end scenarios involving two endpoints.
5. `Assertions` — high-signal result reporting that turns observations into concrete pass/fail diagnoses.

This keeps transport handling, call state, RTP observation, and scenario orchestration independent so failures can be localized cleanly.

## Suggested layout

- `backend/src/bin/headless_call_tester.rs` — CLI entry point
- `backend/src/test_client/mod.rs` — module exports
- `backend/src/test_client/endpoint.rs` — `SipEndpoint`
- `backend/src/test_client/dialog.rs` — `DialogController`
- `backend/src/test_client/rtp_probe.rs` — `RtpProbe`
- `backend/src/test_client/scenario.rs` — scenario orchestration
- `backend/src/test_client/assertions.rs` — shared assertions and failure rendering

The tester stays separate from production server modules. It may reuse generic helpers where practical, but it must not tightly couple itself to server internals that would make it impossible to point the tester at an external SIP3 deployment.

## Runtime modes

### Local / CI mode

Targets a locally running SIP3 stack, typically started with Docker Compose. This mode is the primary regression harness and is required to be deterministic enough for CI.

Inputs:

- SIP domain or host
- TLS port
- two test accounts and passwords
- timeouts
- scenario selection

Expected use:

```text
cargo run --bin headless_call_tester -- --target local --scenario tls-basic-call
```

### Probe mode

Targets a real SIP domain such as `sip.air32.cn` and runs the same core scenarios with tighter safety controls and clearer output for operational diagnosis. This mode is for smoke probing, not for destructive load testing.

Expected use:

```text
cargo run --bin headless_call_tester -- --target sip.air32.cn --scenario tls-basic-call
```

Both modes use the same endpoint, dialog, and RTP logic. Only configuration defaults and reporting context differ.

## Endpoint model

Each endpoint behaves like a minimal TLS SIP softphone:

- opens and maintains a SIP/TLS connection;
- performs REGISTER with digest authentication;
- tracks registration success/failure and expiry;
- can send and receive MESSAGE;
- can originate or answer INVITE dialogs;
- can send ACK, BYE, and CANCEL;
- can expose received SIP events to the scenario runner.

The client does not need a full general-purpose SIP stack in v1. It only needs enough transaction handling for the fixed scenarios. That means a narrow implementation with explicit support for the methods under test is preferred over a broad but weak abstraction.

## Call flow model

The tester models a two-party call with `caller` and `callee`.

1. Both endpoints REGISTER over TLS.
2. Caller sends INVITE with plain RTP SDP generated by the tester.
3. Callee receives INVITE and auto-answers with `180 Ringing` followed by `200 OK`.
4. Caller must receive `200 OK` and send ACK.
5. Both sides start RTP probes against the negotiated SDP targets.
6. Caller ends the call with BYE.
7. Callee must receive BYE and both endpoints must return to idle.

The state machine must distinguish key failures:

- INVITE not delivered to callee
- provisional response missing
- `200 OK` missing on caller side
- ACK not sent after answer
- RTP only one-way
- BYE forwarded but remote dialog not cleared
- caller cancels because answer never arrived

## RTP probe design

`RtpProbe` is intentionally minimal. It does not decode audio and does not attempt MOS, waveform validation, or codec quality checks. Its job is to confirm that SIP3 built a viable media path.

Behavior:

- bind a local UDP port;
- advertise that port in local SDP;
- send fixed-size RTP packets at a stable interval after call establishment;
- count inbound packets;
- record peer source address and packet totals;
- expose metrics back to assertions.

For a scenario to pass, both caller and callee must observe inbound RTP above a minimum packet threshold within a bounded interval. This directly catches the “call connects but no audio” class of regressions without introducing audio-content complexity in v1.

## Fixed scenarios for v1

### `tls_register_dual`

Both endpoints register successfully over TLS.

Assertions:

- both receive `200 OK`;
- both end in a registered state.

### `tls_message_dual`

Two-way MESSAGE exchange between the endpoints.

Assertions:

- 1001 receives MESSAGE from 1003;
- 1003 receives MESSAGE from 1001;
- each MESSAGE transaction receives a successful SIP response.

### `tls_call_setup`

Caller places a call and callee auto-answers.

Assertions:

- callee receives INVITE;
- caller receives `180 Ringing` or equivalent provisional response;
- caller receives `200 OK`;
- caller sends ACK.

### `tls_rtp_bidirectional`

After answer and ACK, both endpoints exchange RTP.

Assertions:

- caller receives RTP from callee above threshold;
- callee receives RTP from caller above threshold;
- source addresses are stable enough to indicate a real bidirectional path.

### `tls_call_teardown`

Caller sends BYE to end the dialog.

Assertions:

- callee receives BYE;
- caller receives success response to BYE;
- both endpoints return to idle;
- no side remains stuck in an active dialog.

## Failure reporting

Pass/fail output must be specific enough to replace live guesswork. Failures must point to the broken boundary, not just the overall scenario name.

Examples:

- `failed: callee did not receive INVITE within 5s`
- `failed: caller did not receive 200 OK within 5s`
- `failed: caller received 200 OK but ACK was not sent`
- `failed: RTP one-way, caller_rx=0 callee_rx=47`
- `failed: BYE reached caller response path but callee dialog did not clear`

Each scenario emits:

- scenario name
- timestamps or elapsed durations
- per-endpoint SIP event summary
- RTP counters
- final verdict

This makes the tool usable both in CI logs and in production-adjacent troubleshooting.

## Configuration

The tester accepts configuration through CLI flags in v1. Config-file support is deferred.

Minimum configuration:

- target host or domain
- TLS port
- caller username/password
- callee username/password
- SIP domain / realm
- scenario name
- per-step timeout values
- RTP packet threshold

The first version assumes exactly two configured accounts. A general-purpose account pool is unnecessary in v1.

## Error handling

- TLS connect failure: fail fast with explicit target/port context.
- REGISTER challenge/auth failure: surface the exact step that failed.
- INVITE timeout: fail call setup without attempting RTP assertions.
- Missing `200 OK`: report as signaling failure, not audio failure.
- RTP threshold miss: report one-way or no-way media separately from signaling success.
- BYE timeout or stuck dialog: report teardown failure even if earlier phases passed.

The tester avoids silent retries that hide state problems. If retries are used at all, they must be explicit in logs and limited to startup conditions such as transient connection establishment.

## Testing strategy

The tester itself is covered at two levels:

1. focused Rust tests for parsing, state transitions, and assertion formatting;
2. scenario-level runs against a local SIP3 stack for end-to-end verification.

The intended regression workflow is:

1. start SIP3 locally;
2. run `headless_call_tester` scenarios in CI or manually;
3. optionally point the same binary at a real domain for smoke probing.

This complements existing backend tests. It does not replace lower-level unit tests around proxying, SDP rewriting, transport registry, or media relay internals.

## Out of scope

- GUI test client
- audio decode or waveform matching
- video
- WebRTC signaling/media
- WSS and UDP transport coverage in v1
- load testing or concurrency stress testing
- generalized scenario DSL

## Future extensions

- add UDP and WSS variants using the same scenario runner
- add WebRTC bridge scenarios
- externalize scenario definitions once the fixed v1 cases are stable
- add richer production probe reporting for scheduled health checks
